Individual services (footer "Leistungen" column, burger menu, "Details ansehen") → their own service page; services without a page yet point to `/#leistungen` for now. Links open in the same window (no `target="_blank"`).
- **Contact CTAs → `/kontakt` everywhere, opening in a NEW TAB** (`target="_blank" rel="noopener"`). **EVERY** contact button site-wide (hero "Angebot anfragen", header/menu CTA, "Anfrage senden", "Objekt besprechen", "Objektbesichtigung anfragen", footer "Kontakt aufnehmen" + footer "Kontakt" nav) points to `/kontakt` — on the service pages **and** on the main page. The on-page contact-panel keeps `id="angebot"` but nothing links to it.
- **`#leistungen` anchor = the 9-service catalog section** (`.services-catalog`), NOT the 3-card overview. The "Leistungen" nav, footer column, burger menu and "alle Leistungen" links all resolve there.
- **Anchor scrolling is smooth + header-offset.** `initSmoothScroll` intercepts same-page anchor clicks and uses `lenis.scrollTo(target, { offset: -(headerHeight + 24) })` (≈104px on desktop) so the fixed desktop header (>1151px) doesn't hide the target heading; it also smooth-scrolls on load when the URL has a hash (e.g. arriving at `/#leistungen` from a service page). Below 1151px the offset is 0 (header scrolls away). **The JS offset is the single source of truth — do NOT also add CSS `scroll-padding-top`**: the two stack and double the gap (was ~208px → fixed to ~104px by removing the CSS rule). Trade-off: reduced-motion users (Lenis off) get no offset, so a cross-page native hash jump can sit the heading under the bar — acceptable edge case.
- **The footer logo is a link to the home page** (`<a href="/">` wrapping the logo img, on every page). `.footer_logo a { display: inline-block; width: fit-content }` keeps the clickable area to the logo only.
- **Build**: `scripts/build-static.mjs` auto-discovers every root-level `.html`, so new pages ship without editing the file list. Images follow the normal pipeline (source → `assets/source/`, `npm run assets:images`, `<picture>` AVIF+WebP, hero img `fetchpriority="high"`, no lazy).

## Current UX Rules
- The main navigation and CTA lead to the contact form.
- Email input blocks spaces immediately and validates format on submit. On blur it also runs the vendored `mailcheck` library to suggest a correction for likely domain typos (e.g. `gmail.con` → `gmail.com`); the suggestion appears as a clickable hint under the field and applies the corrected address on click. Domain lists are Germany-first (web.de, gmx.de, t-online.de, etc.).
- Phone input uses a Germany-first country selector. intl-tel-input owns the field: `strictMode` (valid characters only + length capped to the country's max), `formatAsYouType`/`formatOnDisplay` (native country separators), `autoPlaceholder: "aggressive"` (example-number mask), and `countrySearch: false` (no search box in the dropdown). The normalized E.164 number is stored in the hidden `name="phone"` field; do not reintroduce manual digit-only stripping in the visible field.
- Contact form backend: frontend submits JSON to `POST /api/contact`. The endpoint validates all inputs server-side, checks the honeypot, rate-limits per IP, rejects duplicate payloads for a short window, and only then calls `CONTACT_WEBHOOK_URL`. Configure `CONTACT_WEBHOOK_URL`, `CONTACT_WEBHOOK_SECRET`, and `CONTACT_TO_EMAIL=losoma@web.de` in Vercel before expecting real delivery. Google Sheets + email can use the Apps Script template in `docs/contact-form-apps-script.js`.
- Turnstile is intentionally not enabled yet. The current anti-spam baseline is honeypot + server validation + rate limit + duplicate protection. Add Turnstile only if spam appears or paid traffic starts; if enabled, also update Datenschutzerklärung/DPA.
- The contact form has a hidden honeypot field for bot filtering.
- **The contact form is ONE shared component (`.contact-form` + its `data-*` hooks).** Every page (the CTA panel on the main + service pages, and the standalone `/kontakt` page) uses the same markup, so name/email/phone/dropdown validation, intl-tel-input and mailcheck behave identically everywhere. Re-themes (e.g. the light `/kontakt` variant) only change colours via a scoped parent class — never fork the markup or the JS.
